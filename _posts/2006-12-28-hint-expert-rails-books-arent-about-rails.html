I got my copy of the Agile Rails book, 2nd edition in the mail today. I didn’t feel any of the excitement that I did when I got the first edition or when I got Rails Recipes. Part of the reason is that I already have the PDF, so I’ve seen the book already. More likely though is just that it’s a boring book.

Don’t get me wrong, the Daves have done an excellent job documenting Rails, and the book certainly doesn’t put you to sleep. It’s just that if you’ve worked with Rails for a little while, there’s nothing new to find in this book. Nothing that’s going to be of use, anyway.

The back of the book says it’s targeted for beginner developers all the way through experts. I certainly agree that it’s accessible to new programmers, and it contains basically everything you could want to know about Rails. So why do I say that there’s nothing new of use to experienced Rails developers?

It’s time to let the cat out of the bag. This isn’t a book review.

Rails Experts Are No Different From You And Me

…at least when it comes to using Rails itself.

I know that I can change the table names of my models. I don’t know exactly how, but I know that I can look in the AR::Base docs and find it. I know that I can use composite primary keys. There’s lots of esoteric stuff in Rails that I know is possible, but that I don’t know how to do off the top of my head. I have a feeling that anyone who spends their time writing Rails apps is in the same boat.

We all know that Rails makes the mundane decisions for you. I laugh when I see a forum post where someone asks how he should structure the directories in his PHP app. Are you kidding me? That means this guy is probably sitting around waiting for replies before he even creates directories. Rails is simple. You decide that you want to write a web app, and Rails takes care of the rest. The only real decision is which database to use, and that’s easily reversible.

Good Rails developers realize that Rails takes away all the bullshit that everyone else deals with. I have stacks and stacks of Java books…Struts, Spring, Hibernate, EJBs, Tomcat, blah blah blah. There is so much literature on the infrastructure of writing web applications. What’s left when you take away the need to understand and configure all that infrastructure?

Rails doesn’t have any expert books because it doesn’t require an expert to use Rails. It takes an expert to develop amazing applications, but there’s nothing special about hooking up a Rails application.

Put simply, the expert way of using Rails is nearly always the same as the beginner way.

An expert-level Rails book would be an expert-level book on building applications. It wouldn’t talk about infrastructure, it wouldn’t talk about scaling, it would talk about one thing - how to build interesting, valuable software. There’s precious little literature on what it means to build valuable software. The only book I’ve read that provides a worthwhile discussion is Domain-Driven Design, and I read a LOT of software books.

My Job Didn’t Go To India!

Rails, easy as it is, will ironically make things terribly difficult on both developers and technical writers. They’re going to learn something that we all intrinsically know, yet have a tendency to deny. Programming is HARD! Syntax and configuration and conceptual weight is tough, but it all pales in comparison to the difficulty of actually solving business problems. That’s where intelligence is needed. That’s where creativity is needed. The intellectual power necessary to create value out of nothing dwarfs the power needed to write some code.

I think a lot of programmers are going to find themselves looking for jobs in the coming years, and not because their jobs are getting sent to India. In fact, fewer and fewer jobs will go to India for the same reason these guys will be out of jobs. In the past, and increasingly less so now, the knowledge of how to configure different frameworks and technologies to play well together was mistaken for the ability to develop. Programming jobs are filled with people who are more admins than developers. With things like Rails obviating the need for these pradministrators, the jobs will go to the guys who know how to build valuable software.

This isn’t some doomsday prophecy. Far from it, we’re poised for a software Renaissance. Infrastructure and technical problems have been solved, giving us the time and freedom to work on interesting things. For a developer, this is a very exciting time.
